## What This Is вЂ” Plain English

The Asset Library is the single location where every generated file lives.

Every caption. Every video clip. Every thumbnail. Every blog post draft. Every press release. Once a file is generated, it goes into the Asset Library. Once it is approved, it stays in the Asset Library. Once it is published, the Asset Library records where it was published, when, and what version.

Without an Asset Library, your campaign assets are scattered: some in a Dropbox folder, some in your phone's camera roll, some in a Downloads folder named "final_v3_REAL_final.mp4." When something goes wrong, you don't know which version was published. When you run a second campaign, you can't find the approved thumbnail from the first one to use as a reference.

**One analogy:** The Asset Library is the master tape vault of a recording studio. Every take is stored. Every mix is labeled. Every release version is documented. Nothing is ever lost. Everything is retrievable.

## What Lives in the Asset Library

The Asset Library stores every file generated by or approved for a MusicWorks campaign:

| Asset Category | Types | Notes |
|---------------|-------|-------|
| Video | Short-form clips (raw), assembled video (final), platform variants | MP4 only |
| Images | Thumbnails, quote cards, story frames, cover crops | PNG preferred; JPG for photos |
| Text | Captions (by platform), blog posts, press releases, email copy, church blurbs | Markdown or plain text |
| Audio | References to human-created master only вЂ” no audio generated here | WAV/FLAC master stored at path only |
| Documents | Devotional guides, production briefs, approval logs | PDF or Markdown |
| Metadata | Asset manifests, campaign records, approval logs | JSON |

## Asset Status Lifecycle

Assets move through these statuses, in order:

```
GENERATING  в†’  READY_FOR_REVIEW  в†’  APPROVED   в†’  SCHEDULED  в†’  PUBLISHED
                                 в†’  REJECTED    (end вЂ” archived)
                                 в†’  REVISION_REQUESTED  в†’  GENERATING (new version)

Additional statuses:
  BLOCKED     вЂ” render job blocked; asset not yet created
  ARCHIVED    вЂ” rejected or superseded; kept for reference
  CORRUPTED   вЂ” file exists but failed quality check; triggers re-render
```

**Status rules:**
- Only the Render Orchestrator can set `GENERATING`
- Only the Asset Library can set `READY_FOR_REVIEW` (when file is confirmed received and valid)
- Only the Approval Queue can set `APPROVED`, `REJECTED`, or `REVISION_REQUESTED`
- Only the Publishing Agent can set `SCHEDULED` or `PUBLISHED`
- Nothing can skip a status вЂ” you cannot go from `READY_FOR_REVIEW` to `PUBLISHED` without passing through `APPROVED`

## Versioning

Every time a revision is made to an asset, the Asset Library creates a new version rather than overwriting the original.

**Why versioning matters:**
- If you approve v2 of a video and then decide v1 was better, you can still access v1
- The approval log references the specific version that was approved вЂ” not just the asset type
- Publishing records show exactly which version was published to which platform
- Legal protection: if a theological concern arises after publication, you can show exactly which version was approved, when, and by whom

**Versioning rules:**
- Version numbers are integers starting at 1
- The `version_history` field on each asset record lists all prior version IDs
- Prior versions are never deleted вЂ” they are set to `ARCHIVED` status
- The Publishing Agent always uses the highest-version APPROVED asset

## Quality Gate вЂ” What the Asset Library Checks Before READY_FOR_REVIEW

Before setting an asset to `READY_FOR_REVIEW`, the Asset Library runs automatic checks:

### Video
- [ ] File is a valid MP4 (not corrupt)
- [ ] Duration is within expected range (В±5 seconds of requested)
- [ ] Aspect ratio matches requested (9:16 for short-form)
- [ ] File size is reasonable (not 0 bytes, not > 500MB)
- [ ] First frame is not black (common sign of corrupt render)
- [ ] Audio track is present (if music is expected in this clip)

### Image (Thumbnail)
- [ ] File is a valid PNG or JPG
- [ ] Resolution matches requested (min 1080x1920 for short-form cover)
- [ ] File is not corrupt (can be opened and has non-zero dimensions)
- [ ] File size under 2MB (platform upload limits)

### Text (Captions, Blog Posts)
- [ ] File is not empty
- [ ] No "[PLACEHOLDER]" strings remaining
- [ ] No "[LINK]" fields without URLs
- [ ] Scripture reference present where expected
- [ ] Character count within platform limits (checked against per-platform specs)

If any check fails в†’ asset status = `CORRUPTED` в†’ Render Orchestrator notified в†’ Re-render triggered automatically.

## Preview Generation

When an asset enters the Asset Library, the system automatically generates a preview version for the Approval Queue. This is what the founder sees вЂ” not the full-resolution file, but a lighter preview.

| Asset Type | Preview Format | Why |
|-----------|---------------|-----|
| Video (MP4) | 720p MP4 preview + first-frame JPG thumbnail | Full-res video may be 100MB вЂ” preview is 10MB |
| Image (PNG) | 800px wide WebP | Fast loading in the Approval Queue interface |
| Text | Formatted HTML preview | Renders markdown as readable formatted text |

Preview URLs are stored in the Asset Record (`preview_url` field) and are what the Approval Queue loads by default. Full-resolution files are only loaded when the founder clicks "View full resolution" or when the Publishing Agent needs the final file.

## Storage Options by Version

### V2 вЂ” Simple (Build First)
Google Drive folder organized per the structure above.
Asset records in Airtable (one record per asset).
Preview URLs are Google Drive share links.
No automatic quality checks вЂ” founder checks manually.

**Cost:** Free (with Drive storage limits) or minimal Airtable + Drive cost.

### V2.5 вЂ” Better
Cloudflare R2 for file storage (object storage, very cheap вЂ” ~$0.015/GB).
Asset records in Airtable or PostgreSQL.
Automatic quality checks via Python (Pillow for images, ffprobe for video).
CDN URLs for fast preview loading.

**Cost:** ~$5/month for typical campaign volume.

### V3 вЂ” Production
AWS S3 or Cloudflare R2 for file storage.
PostgreSQL for asset metadata.
Automatic quality checks on ingest.
CDN-distributed preview URLs with signed expiry (security).
Automatic thumbnail generation on video ingest.

## Critical Rules

**Rule 1: Never overwrite. Always version.**
A file in the Asset Library is never modified after it is stored. Revisions create new versions. The original is always accessible.

**Rule 2: The approval log is part of the Asset Library.**
Every approval decision is stored alongside the asset it applies to. The approval log is not separate вЂ” it is attached to each asset record and to the campaign manifest.

**Rule 3: Publishing records are permanent.**
Once an asset is published, the publishing record is never removed. If a post is deleted from a platform, the Asset Library still records that it was published, when, and where. This is the audit trail.

**Rule 4: Audio masters are pointers, not files.**
The Asset Library stores the path to the human-created audio master. It does not own the audio master. The audio master lives in a protected location and is never modified by any automated process.
